## 二、安全漏洞问题

### 2.1 后端安全漏洞

#### SEC-B-01: 搜索关键词 LIKE 通配符注入 ✅ 已修复
- **文件**: `smarttable-backend/app/services/record_service.py` L471-482
- **风险等级**: 🔴 高 → ✅ 已修复
- **问题描述**: `search_records` 方法中，用户提供的搜索关键词 `query` 直接通过 f-string 拼接到 `ilike` 过滤器中 (`f'%{query}%'`)。用户可以通过注入 `%` 和 `_` 等 SQL LIKE 通配符操控搜索行为，导致信息泄露或 DoS 攻击。此外，`field_ids` 参数直接作为 JSON 字段键访问，可能产生意外行为。
- **修复状态**: 已添加 `_escape_like_pattern()` 函数转义 LIKE 通配符，并验证 `field_ids` 为合法 UUID 格式。
- **修复建议**: ~~对 `query` 中的 LIKE 通配符进行转义（`%` → `\%`, `_` → `\_`），并验证 `field_ids` 为合法 UUID 格式。~~ 已完成。

#### SEC-B-02: 缓存键使用 MD5 哈希
- **文件**: `smarttable-backend/app/services/formula_service.py` L1579-1587
- **风险等级**: 🟡 低
- **问题描述**: `_build_cache_key` 方法使用 MD5 哈希构建缓存键，MD5 存在碰撞漏洞，虽在缓存键场景风险较低，但不符合安全最佳实践。
- **修复建议**: 替换为 SHA-256。

#### SEC-B-03: 开发环境硬编码密钥可能泄露到生产环境
- **文件**: `smarttable-backend/app/config.py` L83-86
- **风险等级**: 🔴 高
- **问题描述**: `DevelopmentConfig` 中硬编码了 `SECRET_KEY`、`JWT_SECRET_KEY`、MinIO 凭据的默认值。如果生产环境未正确设置环境变量，应用可能使用弱密钥运行，导致 JWT 令牌可被伪造、存储系统未授权访问。
```python
SECRET_KEY = os.environ.get('SECRET_KEY') or 'dev-secret-key-not-for-production'
JWT_SECRET_KEY = os.environ.get('JWT_SECRET_KEY') or 'dev-jwt-secret-not-for-production'
MINIO_ACCESS_KEY = os.environ.get('MINIO_ACCESS_KEY') or 'minioadmin'
MINIO_SECRET_KEY = os.environ.get('MINIO_SECRET_KEY') or 'minioadmin'
```
- **修复建议**: 移除 `or` 后的默认值，改为在未设置时抛出异常。

#### SEC-B-04: JWT 令牌版本校验未在认证装饰器中执行
- **文件**: `smarttable-backend/app/utils/decorators.py` L101-170
- **风险等级**: 🟠 中
- **问题描述**: `authenticate` 装饰器验证了 JWT 令牌有效性，但未检查令牌版本号。`AuthService` 实现了 `revoke_all_user_tokens` 方法，但认证装饰器从未调用 `validate_token_version`，密码修改后旧令牌仍然有效。
- **修复建议**: 在 `authenticate` 装饰器中添加令牌版本校验逻辑。

#### SEC-B-05: 附件端点缺少资源级权限检查
- **文件**: `smarttable-backend/app/routes/attachments.py` L90-238
- **风险等级**: 🔴 高
- **问题描述**: 附件的获取、下载、预览端点仅检查用户是否登录（`@jwt_required`），未检查用户是否有权访问该附件所属的资源。代码中有 TODO 注释确认了此问题。任何已登录用户可通过枚举 `attachment_id` 访问系统中任意附件，造成水平越权。
- **修复建议**: 实现附件到 Base 的关联，并在访问时检查用户对 Base 的查看权限。

#### SEC-B-06: 删除附件仅检查上传者，管理员无法删除
- **文件**: `smarttable-backend/app/routes/attachments.py` L267-281
- **风险等级**: 🟠 中
- **问题描述**: 删除附件时仅检查当前用户是否为上传者，管理员或 Base 所有者无法删除。如果 `uploaded_by` 为 None，则无人能删除。
- **修复建议**: 增加管理员和 Base 所有者的删除权限检查。

#### SEC-B-07: 验证邮箱端点缺少速率限制
- **文件**: `smarttable-backend/app/routes/auth.py` L616-655
- **风险等级**: 🟡 低
- **问题描述**: `/verify-email` 端点没有速率限制，攻击者可暴力枚举验证令牌。
- **修复建议**: 为该端点添加 `@rate_limit` 装饰器。

#### SEC-B-08: MinIO 凭据硬编码 ✅ 已修复
- **文件**: `smarttable-backend/app/config.py` L85-86
- **风险等级**: 🔴 高 → ✅ 已修复
- **问题描述**: 开发环境配置中硬编码了 MinIO 默认凭据 `minioadmin`/`minioadmin`。
- **修复状态**: 已移除硬编码默认凭据，改为在 `DevelopmentConfig.init_app()` 中检查并发出警告日志。未配置时 MinIO 功能将不可用。
- **修复建议**: ~~移除默认凭据值，改为必须通过环境变量设置。~~ 已完成。

#### SEC-B-09: .env.example 包含示例凭据 ✅ 已修复
- **文件**: `smarttable-backend/.env.example` L7, L17, L21-23
- **风险等级**: 🟡 低 → ✅ 已修复
- **问题描述**: `.env.example` 中包含占位符密钥，可能被开发者直接复制使用而不修改。
- **修复状态**: 已添加醒目的安全警告注释（⚠️），并提供密钥生成方法说明。
- **修复建议**: ~~添加醒目注释，明确标注这些值必须替换为强随机值。~~ 已完成。

#### SEC-B-10: JWT 认证日志记录 Authorization 头
- **文件**: `smarttable-backend/app/utils/decorators.py` L121-122
- **风险等级**: 🟠 中
- **问题描述**: 认证装饰器将 Authorization 头的前 50 个字符写入日志，JWT 令牌的前 50 个字符可能包含足够信息用于令牌伪造。
- **修复建议**: 仅记录令牌是否存在，不记录令牌内容。

#### SEC-B-11: 密码修改日志中记录用户邮箱
- **文件**: `smarttable-backend/app/services/auth_service.py` L300-308
- **风险等级**: 🟡 低
- **问题描述**: 密码修改失败时，操作日志中记录了用户邮箱，如果日志被未授权访问可能泄露用户信息。
- **修复建议**: 在日志中仅记录用户 ID。

#### SEC-B-12: API 文档页面 render_template_string 转义风险
- **文件**: `smarttable-backend/app/__init__.py` L515-774
- **风险等级**: 🟡 低
- **问题描述**: API 文档页面使用 `render_template_string` 渲染 HTML，Jinja2 默认自动转义在 `render_template_string` 中可能不生效。
- **修复建议**: 确保所有动态内容使用 `{{ var|e }}` 进行转义，或使用独立模板文件。

#### SEC-B-13: CSP 策略允许 unsafe-inline 和 unsafe-eval
- **文件**: `smarttable-backend/app/middleware/security_headers.py` L65-77
- **风险等级**: 🟠 中
- **问题描述**: Content-Security-Policy 中 `script-src` 包含 `'unsafe-inline'` 和 `'unsafe-eval'`，大幅削弱了 CSP 对 XSS 攻击的防护效果。
- **修复建议**: 移除 `'unsafe-inline'` 和 `'unsafe-eval'`，改用 nonce 或 hash 方式加载内联脚本。

#### SEC-B-14: API 端点缺少 CSRF 保护
- **文件**: 全局
- **风险等级**: 🟠 中
- **问题描述**: 应用安装了 Flask-WTF 但 API 使用 JWT Bearer Token 认证，CSRF 防护依赖 JWT 机制。部分端点（如静态文件服务）不需要认证，可能被利用。
- **修复建议**: 对所有状态变更操作确保强制要求 JWT 认证，考虑为非 API 页面添加 CSRF Token 保护。

#### SEC-B-15: 静态文件端点无需认证
- **文件**: `smarttable-backend/app/routes/attachments.py` L421-466
- **风险等级**: 🔴 高
- **问题描述**: `serve_uploaded_file` 端点没有 `@jwt_required` 装饰器，任何人都可以通过猜测 URL 路径访问上传的文件。虽有路径遍历检查，但缺少认证是严重安全问题。
- **修复建议**: 为该端点添加认证检查，或使用不可猜测的文件名配合签名 URL。

#### SEC-B-16: .js/.html 扩展名策略矛盾
- **文件**: `smarttable-backend/app/services/attachment_service.py` L69-88
- **风险等级**: 🟠 中
- **问题描述**: `.js` 同时出现在允许和禁止列表中，`.html` 在允许列表但不在禁止列表中，上传的 HTML 文件可能包含恶意脚本。
- **修复建议**: 将 `.html` 添加到 `DANGEROUS_EXTENSIONS`，从允许列表中移除 `.js`。

#### SEC-B-17: 文本文件内容验证过于宽松
- **文件**: `smarttable-backend/app/services/attachment_service.py` L140-156
- **风险等级**: 🟠 中
- **问题描述**: 对文本文件的 Magic Number 验证仅检查是否为有效编码，不检查内容是否包含恶意代码。
- **修复建议**: 对 HTML 文件进行内容扫描或直接禁止上传。

#### SEC-B-18: Record Schema 验证过于宽松
- **文件**: `smarttable-backend/app/schemas/record_schema.py` L1-30
- **风险等级**: 🟠 中
- **问题描述**: `RecordCreateSchema` 仅验证 `values` 字段存在且为 Dict 类型，不验证字典内容，攻击者可提交任意键值对。
- **修复建议**: 添加对 `values` 字典的键（UUID 格式）和值（长度限制、类型检查）的验证。

#### SEC-B-19: 批量操作缺少数量下限验证
- **文件**: `smarttable-backend/app/routes/records.py` L594-599
- **风险等级**: 🟡 低
- **问题描述**: 批量更新端点的 `BatchUpdateSchema` 没有验证 `record_ids` 列表不为空。
- **修复建议**: 在 Schema 中添加 `validate=validate.Length(min=1)` 验证。

#### SEC-B-20: 多个依赖存在已知安全漏洞
- **文件**: `smarttable-backend/requirements.txt`
- **风险等级**: 🟠 中
- **问题描述**: Flask 3.0.0、SQLAlchemy 2.0.23、Pillow 10.4.0、eventlet 0.36.1、gunicorn 21.2.0、flasgger 0.9.7b2 等依赖存在已知安全漏洞或为 Beta 版本。
- **修复建议**: 运行 `pip audit` 检查所有依赖的已知漏洞，并升级到安全版本。

#### SEC-B-21: 异常中泄露内部错误信息
- **文件**: `records.py` L132/L203, `admin.py` L114, `auth.py` L460 等多处
- **风险等级**: 🟠 中
- **问题描述**: 多个路由端点在异常处理中直接将 `str(e)` 返回给客户端，可能泄露内部实现细节、数据库结构、文件路径等敏感信息。
- **修复建议**: 生产环境中 500 错误应返回通用消息，详细错误仅记录到日志。

### 2.2 前端安全漏洞

#### SEC-F-01: RichTextField.vue 使用自定义净化器而非 DOMPurify
- **文件**: `smart-table/src/components/fields/RichTextField.vue` L244, L9, L55-57
- **风险等级**: 🟠 中
- **问题描述**: `v-html="sanitizedHtml"` 使用的是自定义 `sanitizeHtml` 函数（来自 `@/utils/helpers`），而非项目已安装的 DOMPurify。自定义净化器容易出现绕过漏洞，如对 `style` 属性中的 `expression()` 缺乏过滤。
- **修复建议**: 统一使用 `@/utils/sanitize` 中基于 DOMPurify 的实现。

#### SEC-F-02: Home.vue v-html 使用局部转义函数
- **文件**: `smart-table/src/views/Home.vue` L902, L1017, L155-165
- **风险等级**: 🟡 低
- **问题描述**: `v-html="highlightText(base.name, searchQuery)"` 中 `escapeHtml` 是组件内局部定义的函数，而非复用标准实现，存在维护不一致风险。
- **修复建议**: 替换为 `@/utils/sanitize` 中的标准实现。

#### SEC-F-03: EmailTemplates.vue DOMPurify 允许 style 属性
- **文件**: `smart-table/src/views/admin/EmailTemplates.vue` L78, L133
- **风险等级**: 🟡 低
- **问题描述**: `EMAIL_TEMPLATE_CONFIG` 允许了 `style` 属性，可能被利用进行 CSS 注入。
- **修复建议**: 对 style 属性值做进一步过滤。

#### SEC-F-04: JWT Token 存储在 localStorage/sessionStorage
- **文件**: `smart-table/src/utils/auth/token.ts` L15-49
- **风险等级**: 🔴 高
- **问题描述**: JWT access_token 和 refresh_token 默认存储在 `localStorage`/`sessionStorage`，均可被 JavaScript 访问。一旦存在 XSS 漏洞，攻击者可直接窃取 token。
- **修复建议**: access_token 使用 httpOnly cookie 存储；refresh_token 更应使用 httpOnly + Secure + SameSite cookie。

#### SEC-F-05: dashboardShareService 绕过 token 工具直接读取 localStorage
- **文件**: `smart-table/src/db/services/dashboardShareService.ts` L143-144
- **风险等级**: 🟠 中
- **问题描述**: 该服务绕过了 `getToken()` 工具函数，直接通过 `localStorage.getItem("access_token")` 读取 token，硬编码了 key 名称。
- **修复建议**: 改用 `getToken()` 工具函数。

#### SEC-F-06: Refresh Token 在请求体中传输
- **文件**: `smart-table/src/services/api/authService.ts` L32
- **风险等级**: 🟠 中
- **问题描述**: Refresh token 通过请求体发送，可能通过浏览器历史记录、服务器日志等途径泄露。
- **修复建议**: 配合后端将 refresh token 改为 httpOnly cookie 机制。

#### SEC-F-07: withCredentials: true 的跨域风险
- **文件**: `smart-table/src/api/client.ts` L25
- **风险等级**: 🟡 低
- **问题描述**: `withCredentials: true` 使得跨域请求自动携带 cookie，如果后端 CORS 配置不当可能导致凭证泄露。
- **修复建议**: 确保后端 CORS 配置为具体域名而非通配符。

#### SEC-F-08: xlsx ^0.18.5 存在已知漏洞
- **文件**: `smart-table/package.json` L32
- **风险等级**: 🔴 高
- **问题描述**: `xlsx@^0.18.5`（SheetJS 社区版）存在 CVE-2024-43227 原型污染漏洞，且该库已停止社区版维护。
- **修复建议**: 迁移到 `exceljs` 或 `xlsx-populate` 等替代库。

#### SEC-F-09: 缺少 CSRF 防护机制
- **风险等级**: 🟠 中
- **问题描述**: 前端未实现任何 CSRF token 机制。虽然当前使用 Bearer Token 认证有一定防御，但 `withCredentials: true` 意味着 cookie 也会被发送。
- **修复建议**: 在请求头中添加 CSRF token，或确保后端完全依赖 Bearer Token 认证。

#### SEC-F-10: 表单验证规则过于简单
- **文件**: `smart-table/src/views/Home.vue` L102-115
- **风险等级**: 🟡 低
- **问题描述**: 创建/编辑多维表格的表单验证仅包含 `required` 和 `min/max` 长度校验，缺少特殊字符、HTML 标签过滤。
- **修复建议**: 增加 pattern 验证规则，过滤 HTML 标签和特殊字符。

#### SEC-F-11: CSS 注入 - widget config 值直接插入 style 属性
- **文件**: `DashboardShare.vue` L432/472/506/588/699/741, `Dashboard.vue` L1556/1598/1633/1715/1841/1887/1915
- **风险等级**: 🟠 中
- **问题描述**: 多处 widget 配置值（`config.backgroundColor`、`config.textColor` 等）直接插入到 `style="..."` 属性中，未经校验或转义，可能突破 style 属性边界注入 HTML 事件处理器。
- **修复建议**: 对 widget config 中的 CSS 值增加白名单校验（仅允许合法颜色值）。

#### SEC-F-12: DashboardShare.vue innerHTML 中字段名未转义

- **风险等级**: 🔴 高
- **问题描述**: `removeEventListeners` 中 `socketClient.value.off('connect', () => {})` 传入了新的匿名函数，而非注册时的函数引用，导致 `connect` 和 `disconnect` 事件监听器永远不会被移除。
- **修复建议**: 将事件处理器保存为变量引用，在 off 时传入相同引用。

#### BUG-F-22: socketClient.disconnect 清除了所有 realtimeEventEmitter 监听器
- **文件**: `smart-table/src/services/realtime/socketClient.ts` L161-174
- **风险等级**: 🔴 高
- **问题描述**: `disconnect()` 调用 `realtimeEventEmitter.removeAllListeners()` 清除所有事件的所有监听器，断开一个 socket 连接会影响其他组件的监听器。
- **修复建议**: 改为按事件名和处理器引用移除特定监听器。

#### BUG-F-23: tableStore 和 viewStore 的 realtime 监听器未在组件卸载时清理
- **文件**: `smart-table/src/stores/tableStore.ts` L327-426
- **风险等级**: 🟠 中
- **问题描述**: `setupRealtimeListeners` 返回清理函数但 store 不会自动调用，`cleanupRealtimeListeners` 使用 `removeAllListeners` 会影响其他组件。
- **修复建议**: 在组件 `onUnmounted` 时调用清理函数，改用精确移除。

#### BUG-F-24: acquireLock 不等待服务器确认
- **文件**: `smart-table/src/composables/useRealtimeCollaboration.ts` L340-358
- **风险等级**: 🔴 高
- **问题描述**: `acquireLock` emit `lock:acquire` 事件后立即 `resolve({ success: true })`，不等待服务器返回 `lock_result` 事件，可能导致多个用户同时编辑同一单元格。
- **修复建议**: 等待服务器返回 `lock_result` 事件后再 resolve。

#### BUG-F-25: recordCount 可能变为负数
- **文件**: `smart-table/src/stores/tableStore.ts` L289-295, L362-368
- **风险等级**: 🟠 中
- **问题描述**: `deleteRecord` 和 `onRecordDeleted` 都会减少 `recordCount`，实时协作下双重减少可能导致负数。
- **修复建议**: 在 `onRecordDeleted` 中检查是否为当前用户操作，避免双重减少。

#### BUG-F-26: createRecord 中 recordCount 可能重复计数
- **文件**: `smart-table/src/stores/tableStore.ts` L225-254
- **风险等级**: 🟠 中
- **问题描述**: `addRecordFromRemote` 方法中仍有 `recordCount++`，如果被其他路径调用会导致重复计数。
- **修复建议**: 确保 `addRecordFromRemote` 不增加 recordCount，或统一计数逻辑。

#### BUG-F-27: reorderViews/reorderTables/reorderFields 可能静默删除数据
- **文件**: `smart-table/src/stores/viewStore.ts` L212-228, `tableStore.ts` L125-138
- **风险等级**: 🟠 中
- **问题描述**: 使用 `filter(Boolean)` 过滤 undefined，当 ID 列表不完整时会导致不在列表中的项被静默删除。
- **修复建议**: 不使用 `filter(Boolean)`，改为在找不到项时保留原位置或抛出错误。

#### BUG-F-28: IndexedDB 数据同步时序问题
- **文件**: `smart-table/src/db/services/recordService.ts` L123-166
- **风险等级**: 🟠 中
- **问题描述**: `getRecordsByTable` 先 API 后本地，`db.records.put()` 会覆盖但不删除 IndexedDB 中 API 未返回的记录，可能包含已删除的记录。
- **修复建议**: API 返回后先清空该表的 IndexedDB 记录再写入。

#### BUG-F-29: batchUpdateRecords 串行调用 API，部分失败时状态不一致
- **文件**: `smart-table/src/db/services/recordService.ts` L278-299
- **风险等级**: 🟠 中
- **问题描述**: 串行调用 API，中间失败后外层直接 throw，调用方无法知道哪些记录更新成功。
- **修复建议**: 返回部分成功结果，或使用后端批量更新接口。

#### BUG-F-30: disconnect 后状态设为 reconnecting 可能导致无限重连
- **文件**: `smart-table/src/composables/useRealtimeCollaboration.ts` L117-119
- **风险等级**: 🟠 中
- **问题描述**: SocketIO 触发 disconnect 时状态设为 `reconnecting`，但未检查断开原因。如果是认证失败，不应自动重连。
- **修复建议**: 检查断开原因，仅在非主动断开时触发重连。

#### BUG-F-31: 路由守卫引用不存在的 store 路径和方法
- **文件**: `smart-table/src/router/guards.ts` L7, L21, L26, L35, L61, L70
- **风险等级**: 🔴 高
- **问题描述**: 从 `@/stores/auth/authStore` 导入但实际路径为 `@/stores/authStore`；使用了 `isLoggedIn`、`checkAuth()`、`hasPermission()` 等不存在的方法。
- **修复建议**: 修正导入路径，使用 `isAuthenticated` 计算属性替代不存在的方法。

#### BUG-F-32: refreshToken 初始化为 accessToken
- **文件**: `smart-table/src/stores/authStore.ts` L10
- **风险等级**: 🔴 高
- **问题描述**: `const refreshTokenValue = ref<string | null>(getToken())` 初始化时获取的值与 access token 相同，`refreshAccessToken` 可能使用 access token 去请求 refresh 接口导致失败。
- **修复建议**: 使用独立的 `getRefreshToken()` 函数获取 refresh token。

#### BUG-F-33: eventEmitter.emit 中异常被静默吞掉
- **文件**: `smart-table/src/services/realtime/eventEmitter.ts` L35-40
- **风险等级**: 🟡 低
- **问题描述**: `emit` 方法中 `catch (_e) {}` 完全吞掉了事件处理器的异常，不记录任何日志。
- **修复建议**: 至少添加 `console.error` 记录异常。

#### BUG-F-34: deleteView 后 currentView 可能引用已删除视图
- **文件**: `smart-table/src/stores/viewStore.ts` L174-184
- **风险等级**: 🟡 低
- **问题描述**: 删除唯一视图后 `currentView` 设为 null，后续代码可能未正确处理。
- **修复建议**: 添加 null 检查或自动创建默认视图。

#### BUG-F-35: baseStore.createBase 中 IndexedDB 写入缺少 id 字段
- **文件**: `smart-table/src/stores/baseStore.ts` L186-210
- **风险等级**: 🟠 中
- **问题描述**: 写入 IndexedDB 时未传 `id`，导致本地与 API 返回的 base 使用不同 ID，后续同步可能冲突。
- **修复建议**: 使用 API 返回的 `newBase.id` 写入 IndexedDB。

#### BUG-F-36: baseStore.fetchBases 缓存回退时 owner_id 为空字符串
- **文件**: `smart-table/src/stores/baseStore.ts` L121-143
- **风险等级**: 🟠 中
- **问题描述**: API 失败回退到 IndexedDB 缓存时 `owner_id` 被设为空字符串，导致权限检查失败。
- **修复建议**: 在 IndexedDB schema 中添加 `owner_id` 字段。
