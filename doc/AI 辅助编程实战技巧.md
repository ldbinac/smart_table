# 跟 AI 结对编程这一年：我踩过的坑和攒下的经验

> 写在前面：这不是什么"AI 颠覆编程"的鸡汤，只是一个普通开发者在真实项目里，和 AI 从互相嫌弃到配合默契的点滴记录。

---

## 一、先说结论：AI 能帮你写代码，但别指望它懂你的项目

用 AI 编程一年，最大的感悟就一条：**AI 是个勤奋但没经验的实习生**。它手速快、知识广、24 小时待命，但对你项目的来龙去脉一无所知。

所以，用 AI 的核心就两件事：
1. **把话说清楚**（提示词要具体）
2. **把成果盯紧了**（每步都要验收）

---

## 二、那些真正管用的技巧

### 1. 先写文档，再写代码

这个习惯救了我无数次。

以前我直接跟 AI 说："帮我写个用户管理功能"。结果它给我的代码要么缺权限校验，要么数据库设计不合理。

现在我会先让 AI 帮忙写两份文档：
- **spec.md**（功能规格说明）：这个功能要做什么、不做什么、边界条件是什么
- **tasks.md**（任务拆解）：分成几个步骤，每步产出什么

**真实案例**：
> 有次要做"超管用户管理"功能，我先让 AI 写了 spec 文档，列出了 12 个具体任务。结果做到第 5 个任务时发现，第 3 个任务的数据库设计漏了个字段。因为文档写得清楚，回退成本很低，改完再继续。

**好处**：
- 逼着自己把需求想清楚
- AI 不容易跑偏
- 出了问题容易定位是哪一步的锅

### 2. 给完整的上下文，别让它猜

AI 最擅长的其实是"补全"，而不是"创作"。

**错误示范**：
```
帮我写个 API 接口，返回用户列表
```

**正确做法**：
```
项目用的是 Flask + SQLAlchemy，现有代码在 app/routes/admin.py
参考现有的 GET /api/admin/users 接口风格，新增一个 PATCH 接口：
- 路径：/api/admin/users/<user_id>/status
- 功能：更新用户状态（active/suspended）
- 需要 @jwt_required 和 @admin_required 装饰器
- 返回格式参考同文件第 120 行的接口
```

**经验值**：
- 附上相关代码文件的链接或片段
- 说明技术栈和框架版本
- 指出可以参考的现有代码

### 3. 三步验证法：别让 AI 蒙混过关

AI 有个毛病：**即使不确定，也会编个看似合理的答案**。

我的验证流程：
1. **让它自圆其说**：写完代码后，问它"这段代码有没有潜在问题？"
2. **让它写测试**：要求写完整的单元测试，跑不通就说明代码有问题
3. **人工验收**：每完成一个功能点，立刻在浏览器里测，别等全部做完

**血泪教训**：
> 有次 AI 写了个分页接口，代码看着没问题，测试也过了。但实际一用，发现它把 `page` 和 `per_page` 参数搞反了。因为我觉得"这种简单逻辑不会错"，就没细看。结果后面十几个接口都按错误的参数写的，回退到凌晨两点。

### 4. 三次法则：尝试三次还不成，就自己动手

这是个很重要的边界判断。

如果一个简单问题，AI 尝试了三次还解决不了，通常是以下情况之一：
- **提示词有问题**：需求描述不够清晰
- **上下文缺失**：AI 看不到关键的依赖代码
- **历史债务**：前面的代码就有问题，只是没发现

**这时候该做的**：
1. 检查提示词，补充缺失的上下文
2. 往前翻，看是不是之前的代码就有 bug
3. **果断回退**，修正提示词重新来
4. 如果还是很简单的需求（比如调个 CSS 样式），**直接自己写**

**哪些适合自己动手**：
- CSS 样式调整（AI 对视觉不敏感）
- 简单的配置修改
- 涉及多个文件的样式覆盖问题
- 需要结合业务理解的逻辑

### 5. 代码规范化要趁早

这个坑我踩得最深。

刚开始用 AI，它生成的代码没有分号。我觉得"反正能跑"，就没管。结果后面每生成一个新功能，都会因为缺少分号导致各种语法错误。

**正确的做法**：
- 第一次发现代码风格问题（比如缺分号、缩进混乱），**立刻让 AI 修正**
- 让它给出一个统一的代码规范，后续都按这个来
- 定期让它"扫描全文，修复风格不一致的地方"

**话术参考**：
```
我发现你生成的代码都没有分号。从现在开始，所有 JavaScript/TypeScript 代码都要：
1. 语句末尾加分号
2. 函数括号前加空格
3. 导入语句按字母顺序排列

请检查之前生成的所有代码，给出一份需要修改的文件列表。
```

## 三、进阶技巧：让 AI 帮你做更多

### 1. 安全扫描：让 AI 当自己的代码审查员

AI 写的代码，安全性是个大问题。我现在的固定流程：

```
请对刚才生成的所有代码进行安全审查，重点关注：
1. SQL 注入风险（有没有直接拼接 SQL）
2. XSS 攻击风险（用户输入有没有转义）
3. 权限校验（有没有漏掉 @jwt_required）
4. 敏感信息泄露（有没有硬编码密码、密钥）

列出所有潜在问题，并给出修复方案。
```

**真实案例**：
> AI 写了个用户查询接口，直接用 f-string 拼接 SQL。我让它做安全扫描时自己发现了，改成了参数化查询。

### 2. 重构要及时：别让技术债务滚雪球

AI 生成的代码有个特点：**重复代码多**。

我现在的做法：
- 每完成 3-5 个功能点，就让 AI 做一次重构
- 重点检查：有没有重复的代码块？能不能抽取公共函数？
- 要求它给出重构前后的对比，确保功能不变

**话术参考**：
```
请扫描 src/services/ 目录下的所有文件，找出：
1. 重复出现的代码块（超过 3 行相同）
2. 可以抽取的公共工具函数
3. 可以合并的相似逻辑

给出重构方案，并说明每个改动的好处。
```

### 3. 用截图说话

有些需求，文字描述真的很累。

**适用场景**：
- UI 样式调整（比如"把这个按钮往右挪一点"）
- 布局问题（比如"这个表格在移动端显示不全"）
- 交互效果（比如"点击后应该弹出这个菜单"）

**做法**：
1. 截个图，圈出问题所在
2. 告诉 AI 期望的效果
3. 让它对比图文理解

**经验值**：
- 用箭头、方框明确标注问题区域
- 如果有参考设计，一起截图给它
- 说明截图的设备、浏览器、分辨率

## 四、这些坑，你别踩

### 1. 别等"全部做完"再验收

**错误做法**：
> "你先写完这 10 个功能，我一起测"

**正确做法**：
> "写完第一个功能就叫我，我马上测"

**原因**：
- 越晚发现问题，回退成本越高
- AI 会基于错误的前提继续写，错上加错
- 小步快跑，随时调整方向

### 2. 别迷信"更强的模型"

模型之间确实有差异，但**提示词的质量 > 模型的选择**。

我用下来的感受：
- 复杂逻辑、需要推理的：选更强的模型（比如 Qwen-Max、GPT-4）
- 简单任务、代码补全：中等模型就够了
- 关键是要把提示词写清楚

### 3. 别让 AI 做它不擅长的事

**AI 擅长的**：
- 写 CRUD 接口
- 生成重复性代码
- 写单元测试
- 代码解释、文档生成

**AI 不擅长的**：
- 理解复杂的业务逻辑
- 做架构设计
- 处理视觉相关的样式问题
- 需要"常识"判断的场景

**我的原则**：
- 业务逻辑复杂的，自己写核心部分，AI 补充边缘代码
- 架构设计自己把握，AI 只给建议
- CSS 样式问题，直接自己调

## 五、最后的一些心里话

### 1. AI 不会取代你，但会用 AI 的人会取代不会用的

这话听着像鸡汤，但真是这么回事。

我现在的工作流：
- 30% 时间：想清楚要做什么（需求分析、设计）
- 40% 时间：跟 AI 沟通（写提示词、审查代码）
- 20% 时间：自己写核心逻辑
- 10% 时间：测试和验收

**效率提升是实打实的**，但前提是**你知道自己在做什么**。

### 2. 保持"怀疑但开放"的心态

- **怀疑**：AI 说的每句话都要验证，不要盲目相信
- **开放**：接受 AI 给出的更好方案，别固执己见

**真实案例**：
> 有次我要写个分页功能，准备用传统的 `LIMIT OFFSET`。AI 建议用"游标分页"，说性能更好。我一开始不信，让它给出性能对比数据，看完后接受了。最后证明在那个场景下确实更好。

### 3. 最重要的能力：判断什么是"好代码"

用 AI 编程，最核心的能力不是写代码，而是**判断代码好坏**。

- 这段代码有没有安全隐患？
- 这个设计会不会有性能问题？
- 这个实现方式可维护吗？
- 这个测试覆盖全面吗？

**这些判断力，AI 给不了你，只能自己积累。**

## 写在最后

跟 AI 结对编程这一年，我最大的感受是：**它不是来取代你的，是来放大你的能力的**。

你用得好，它能帮你省下大量重复劳动，让你专注在更有价值的事情上。你用不好，它会给你制造一堆技术债务，让你加班到深夜。

关键不在于 AI 有多强，而在于**你是否清楚自己在做什么**。

共勉。
